# Database (PostgreSQL)

## Overview

The **PostgreSQL database** is the central data layer of the F10.7 Solar Flux Forecasting System.  
It stores model configurations, feature definitions, daily feature data, and all prediction outputs.  
All microservices communicate through this shared database to maintain consistency across model inputs, outputs, and metadata.

---

## Architecture

```
Spring Boot Backend
       │
       ▼
  PostgreSQL Database
       │
 ├── model_registry       → list of registered models
 ├── model_features       → defines each model’s input features
 ├── feature_catalog      → metadata for all available features
 ├── features_daily       → computed feature values (from fetch-service)
 ├── predictions          → stored model predictions
 └── ground_truths        → actual observed solar flux values
```

---

## Key Tables

### 1. `model_registry`

Stores information about each machine learning model available in the system.

| Column        | Type        | Description                               |
| ------------- | ----------- | ----------------------------------------- |
| `model_id`    | TEXT (PK)   | Unique identifier for the model           |
| `family`      | TEXT        | Model type (e.g., LightGBM, RandomForest) |
| `description` | TEXT        | Optional notes or details about the model |
| `created_at`  | TIMESTAMPTZ | Timestamp of model registration           |

Example:

```sql
SELECT * FROM model_registry;
```

---

### 2. `model_features`

Links each model to the feature names it expects as input.

| Column         | Type      | Description                         |
| -------------- | --------- | ----------------------------------- |
| `model_id`     | TEXT (FK) | References model_registry(model_id) |
| `feature_name` | TEXT (FK) | References feature_catalog(name)    |

Example:

```sql
SELECT * FROM model_features WHERE model_id = 'lgb_f107_lag27_ap_lag3';
```

---

### 3. `feature_catalog`

Defines and describes all potential features used in any forecasting model.

| Column           | Type           | Description                           |
| ---------------- | -------------- | ------------------------------------- |
| `feature_id`     | BIGSERIAL (PK) | Unique identifier                     |
| `name`           | TEXT           | Feature name (e.g., f107_lag_1)       |
| `source`         | TEXT           | Data origin (e.g., NOAA_F107)         |
| `transformation` | TEXT           | Feature transformation (e.g., lag(1)) |
| `description`    | TEXT           | Human-readable explanation            |
| `created_at`     | TIMESTAMPTZ    | Timestamp of feature registration     |

Example:

```sql
SELECT * FROM feature_catalog LIMIT 5;
```

---

### 4. `features_daily`

Contains the **actual numeric values** used as model input features for each date.

| Column                        | Type             | Description                 |
| ----------------------------- | ---------------- | --------------------------- |
| `date`                        | DATE (PK)        | Date of observation         |
| `flux`                        | DOUBLE PRECISION | Raw flux value              |
| `target_flux`                 | DOUBLE PRECISION | 7-day ahead flux target     |
| `f107_lag_*`                  | DOUBLE PRECISION | F10.7 lag features (1–27)   |
| `ap_mean_lag*`, `ap_max_lag*` | DOUBLE PRECISION | Geomagnetic Ap lag features |

Example:

```sql
SELECT date, flux, f107_lag_1, ap_mean_lag1 FROM features_daily ORDER BY date DESC LIMIT 5;
```

---

### 5. `predictions`

Stores all predictions generated by the model-service.

| Column            | Type             | Description                                 |
| ----------------- | ---------------- | ------------------------------------------- |
| `id`              | BIGSERIAL (PK)   | Unique identifier                           |
| `requested_at`    | TIMESTAMPTZ      | When the prediction was made                |
| `prediction_date` | DATE             | The date the prediction was generated       |
| `horizon_days`    | INT              | Forecast horizon (e.g., 1 for 1-day ahead)  |
| `target_date`     | DATE             | The target day being predicted              |
| `predicted_value` | DOUBLE PRECISION | Predicted F10.7 flux                        |
| `model_id`        | TEXT (FK)        | References model_registry                   |
| `features`        | JSONB            | Serialized features used for the prediction |
| `notes`           | TEXT             | Optional notes or comments                  |

Example:

```sql
SELECT model_id, predicted_value, target_date FROM predictions ORDER BY requested_at DESC LIMIT 10;
```

---

### 6. `ground_truths`

Stores actual observed solar flux measurements used to evaluate prediction accuracy.

| Column             | Type             | Description                        |
| ------------------ | ---------------- | ---------------------------------- |
| `observation_date` | DATE (PK)        | Date of actual observation         |
| `actual_flux`      | DOUBLE PRECISION | Measured F10.7 flux                |
| `source_meta`      | JSONB            | Additional metadata or source info |

Example:

```sql
SELECT * FROM ground_truths ORDER BY observation_date DESC LIMIT 5;
```

---

## Initialization

The database schema is automatically created and populated when the Postgres container starts using the script:

```
db/init/001_schema.sql
```

This script:

- Creates all core tables.
- Inserts feature metadata and model definitions.
- Adds initial index constraints for optimized queries.

---

## Accessing the Database

### Via Docker

```bash
docker exec -it postgres-spring-boot bash
psql -U aspteam -d forecastdb
```

### Common Commands

```sql
\dt                                -- list all tables
SELECT COUNT(*) FROM predictions;  -- count predictions
SELECT * FROM model_registry;      -- view registered models
```

### Reset Database

```bash
docker compose down -v
docker compose up db
```

This will remove all existing data and reinitialize the schema from `001_schema.sql`.

---

## Maintenance Notes

- `feature_catalog` can be extended safely to include new feature types as new models are introduced.
- `model_registry` entries must be created **before** inserting into `model_features`.
- When retraining a model, ensure its `model_id` and `model_features` list are consistent with the `.pkl` file used in the model-service.
